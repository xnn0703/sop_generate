# CLAUDE.md — sop_generate 项目硬性规则

本文件给 Claude Code 看，定义本项目的**强制规定**，每次对项目做改动都必须遵守。

---

## 🔴 硬性规定 R1：使用说明必须与代码同步

**任何用户可见的改动，都必须在同一次提交里同步更新 `docs/使用说明.html`。**

### 1.1 什么算"用户可见改动"

只要属于以下任一项，就**必须**同步使用说明：

| 改动类型 | 举例 |
|---------|------|
| 工具栏按钮 | 新增/删除/改名/换图标 |
| 菜单或快捷键 | 任何变化 |
| 字段限制 | 数量上限、长度限制、字符集 |
| 字段名称 | 工序名、操作说明等显示文字 |
| 数据目录结构 | `sop_packages/` 子目录的组织方式 |
| 文件命名规则 | 输出文件、归档文件、.sopkg 等 |
| 弹窗文案 / 错误信息 | 用户能看到的所有文字 |
| 新增 / 移除功能 | 自动更新、归档、追溯、迁移等 |
| 版本号 / 软件标题 | `core/__init__.py` 的 `__version__` |
| 发版仓库地址 | `release.config.json` 或 CI 配置中的 owner/repo |

### 1.2 不算"用户可见改动"

下面这些只改代码、不改用户感知，**不需要**动使用说明：
- 内部重构（函数拆分、变量改名等）
- 性能优化（不改用户行为）
- bug 修复（行为回到说明描述的样子，说明本来就对）
- CI workflow 内部步骤调整（不影响产物）
- 注释、日志、调试代码

### 1.3 改使用说明的位置

使用说明文件：**`docs/使用说明.html`**（单文件，所有 CSS 内嵌）

涉及主界面截图时，截图放在 **`docs/images/`**，引用相对路径 `images/xxx.png`。

侧边导航在文件顶部 `<aside class="sidebar">`，章节锚点用 `<section id="...">`。新增章节别忘了：
1. 加 `<section id="新锚点">` 内容
2. 侧边导航 `<a href="#新锚点">` 加链接
3. 顶部"📖 本文档章节"快查（如果是用户高频章节）

### 1.4 提交规范

凡是触发本规则的提交，commit message 里**明确写出**同步了说明：

```
feat: 新增"归档定稿"功能

- 工具栏加 📦 归档定稿 按钮
- 自动生成 HTML + PDF + .sopkg 包（文件名带日期）
- docs/使用说明.html 同步新增第 13 章
```

**禁止**在没动 `docs/使用说明.html` 的情况下提交"用户可见改动"——这种 PR/commit 一律视为不完整。

### 1.5 检查清单

每次准备 commit 前自查：

- [ ] 这次改动有没有改用户看到的东西？（按 1.1 表格逐项过）
- [ ] 如果有，`docs/使用说明.html` 是否已同步？
- [ ] 改动涉及版本号 → CHANGELOG.md 也加条目了吗？
- [ ] 改动涉及主界面 → 截图 `docs/images/main-ui.png` 是否要重拍？

## 🟡 软性约定 R2：发版前更新 CHANGELOG.md

每次准备打 v* tag 前，`CHANGELOG.md` 顶部必须新增对应版本的条目，说明：
- 新功能
- Bug 修复
- 破坏性变更（如有）
- 升级注意事项（特别是不能自动升级的情况）

---

## 🟡 软性约定 R3：默认值不带业务信息

`gui/model.py` 的 `DEFAULT_PRODUCT` 和 `products/_schema.yaml`（或 `sop_packages/_schema.yaml`）里：

- ❌ 不能写真实公司名、产品代号
- ✅ 用占位："公司名称"、"DOC-XX-001"、"DEMO01" 等

理由：这是个工具，会发给客户/外部使用，业务信息不应该内嵌。

---

## 🟡 软性约定 R4：跨平台路径

涉及路径的代码必须用 `pathlib.Path`，不能用裸字符串拼接。
- ✅ `app_dir() / "sop_packages" / model / "product.yaml"`
- ❌ `app_dir + "/sop_packages/" + model + "/product.yaml"`

并且：路径解析统一走 `core/paths.py` 提供的函数，**不要**在其他文件里 hardcode 子目录名。

## 🟢 工作流偏好（参考 ~/.claude/CLAUDE.md 全局规则）

- 修改代码后用 `build.sh` 或 venv 中的脚本构建，优先用脚本而不是手动 cmake
- 不要主动 `git commit`，除非用户明确要求
- 每次功能开发前先 plan.md 明确验收标准，用户确认后再实施
- 回答用中文，代码和命令输出保留原文
